# AKQuant 产品需求与架构设计文档

> 版本：0.1
> 日期：2026-06-08
> 范围：仅输出 PRD 与架构设计，不包含代码实现。

## 1. 执行摘要

AKQuant 是一个面向 A 股量化研究的“数据-策略-回测-归因”工作台。它的核心目标不是再做一个只展示收益曲线的回测工具，而是把 A 股本土交易规则、数据质量控制、成本建模和回测后归因分析内化到研究流程中。

建议第一阶段聚焦 V0.1 MVP，先验证一条完整、可信、可复现的主链路：

```text
选择标的 -> 加载/缓存行情数据 -> 运行策略 -> 应用 A 股规则撮合 -> 计算绩效指标 -> 输出可审计报告
```

产品早期不追求功能面大而全，应优先证明以下三件事：

1. 回测结果不会因为忽略 T+1、整手交易、印花税、滑点等规则而明显乐观。
2. 每次回测都有可追溯的数据、参数、成本、订单和成交记录。
3. 报告能够解释“赚了多少、为什么赚、风险在哪、结果是否可靠”。

本产品仅定位于研究与回测分析，不提供实盘自动交易、券商接口、下单执行或投资建议。

## 2. 产品定位

### 2.1 一句话定位

一个面向 A 股的研究、回测与归因分析工作台，帮助用户在真实市场约束下快速验证策略想法，并形成可解释、可复现的研究结论。

### 2.2 目标用户

| 用户类型 | 核心需求 | 产品适配度 |
| --- | --- | --- |
| 个人量化研究者 | 低成本获取 A 股数据，快速验证策略 | 高 |
| 金融/计算机专业学生 | 学习策略研究、回测与风险分析 | 高 |
| 私募/资管初级研究员 | 形成可复现的研究报告和策略对比结果 | 中高 |
| 专业交易团队 | 实盘执行、机构数据、组合风险系统 | 早期不覆盖 |

### 2.3 核心痛点

1. A 股数据源分散，接口稳定性、字段定义、复权口径经常不一致。
2. 通用回测工具往往默认海外市场规则，容易忽略 A 股 T+1、涨跌停、停复牌、整手交易、印花税等约束。
3. 很多策略研究只输出收益曲线，缺少交易行为、成本、回撤、归因和稳健性分析。
4. 回测容易出现幸存者偏差、未来函数、过拟合和成本低估。
5. 研究流程碎片化，数据获取、参数配置、结果输出和报告归档难以复现。

### 2.4 差异化

AKQuant 的差异化来自三层：

1. **数据本土化**：以 A 股行情、指数、行业、资金流、停复牌、涨跌停等数据为核心。
2. **规则本土化**：回测引擎默认支持 A 股交易规则和成本模型。
3. **分析本土化**：报告不只展示收益，还解释选股、择时、行业、风格、成本和交易行为。

### 2.5 产品边界

范围内：

- 历史行情数据获取和缓存。
- 策略回测。
- A 股交易规则模拟。
- 绩效、风险、交易行为和归因分析。
- 本地研究报告生成。
- 多策略和参数结果对比。

范围外：

- 实盘自动交易。
- 券商 API 对接。
- 投资建议生成。
- 高频 tick 级撮合。
- 机构级组合风控系统。

## 3. 产品目标与非目标

### 3.1 产品目标

| 目标 | 描述 | 成功信号 |
| --- | --- | --- |
| 回测更真实 | 默认模拟 T+1、整手、成本、滑点、涨跌停等规则 | 加入规则后结果与裸回测有可解释差异 |
| 研究可复现 | 每次运行保存配置、数据范围、复权方式、成本参数和输出结果 | 同一配置和数据快照能复现同一结果 |
| 结果可解释 | 报告展示收益、风险、成本、交易行为和归因 | 用户能判断收益来源和主要风险 |
| 迭代速度快 | 使用本地缓存和统一数据接口 | 重复回测不反复请求外部接口 |
| 默认保守 | 成本和滑点默认开启 | 用户不会无意中跑出零成本乐观回测 |

### 3.2 非目标

- V0.1 不做完整平台化策略插件市场。
- V0.1 不做复杂 UI，而是先保证回测主链路可靠。
- V0.1 不做因子平台全量能力。
- V0.1 不处理高频或盘口级交易。
- V0.1 不覆盖所有 A 股特殊交易制度，只覆盖最影响结果的基础规则。

## 4. 版本规划

### 4.1 V0.1 MVP：验证主链路

V0.1 的目标是跑通一个可信的单策略回测流程。

必备能力：

1. 支持单标的或少量标的的日线行情加载。
2. 支持前复权、后复权或不复权口径记录。
3. 支持一个内置策略模板。
4. 支持基础 A 股规则：
   - T+1。
   - 100 股整手交易。
   - 买卖手续费。
   - 卖出印花税。
   - 可配置滑点。
   - 涨跌停近似限制。
5. 输出基础绩效指标：
   - 累计收益。
   - 年化收益。
   - 年化波动率。
   - 夏普比率。
   - 最大回撤。
   - 胜率。
   - 换手率。
   - 交易成本。
6. 输出订单、成交、持仓、资金曲线和交易明细。
7. 保存运行配置和报告。

推荐 V0.1 内置策略：

- 双均线趋势策略。
- 原因：逻辑简单、用户熟悉、容易测试，并且对交易成本和 T+1 约束敏感。

### 4.2 V0.5：可用研究工作台

在主链路稳定后扩展研究能力：

- 多标的组合回测。
- 股票池配置。
- 策略库：
  - 双均线。
  - MACD。
  - 布林带。
  - RSI。
  - 多因子排序。
  - 行业轮动。
  - 指数择时。
- 参数网格搜索。
- 策略横向对比。
- 本地缓存管理。
- Streamlit 或 Dash 交互界面。

### 4.3 V1.0：形成差异化

增强专业分析能力：

- 因子 IC/IR。
- 因子分层回测。
- 因子衰减。
- 行业贡献分解。
- 选股 vs 择时归因。
- 风格暴露分析。
- 北向资金、龙虎榜、行业资金流、涨跌停等 A 股特色看板。

### 4.4 V1.5：稳健性分析

重点防止过拟合：

- 样本内/样本外切分。
- 滚动回测。
- 参数高原检测。
- 多市场阶段分段表现。
- 策略稳定性评分。
- 参数敏感性热力图。

### 4.5 V2.0：平台化

面向可扩展研究平台：

- 策略插件机制。
- Notebook 集成。
- 多策略组合管理。
- 研究任务队列。
- 数据源扩展机制。
- 团队共享报告。

## 5. 用户流程

### 5.1 策略研究流程

1. 用户选择策略模板。
2. 用户输入标的或股票池。
3. 用户选择回测区间。
4. 用户设置策略参数。
5. 用户设置初始资金、手续费、滑点和基准。
6. 系统检查本地缓存。
7. 系统拉取缺失数据并做标准化。
8. 系统执行数据质量校验。
9. 系统运行策略并生成订单。
10. 市场规则层校验订单。
11. 撮合层生成成交和拒单记录。
12. 系统更新持仓、现金和组合净值。
13. 系统生成指标、图表和报告。
14. 用户阅读报告并决定是否继续优化。

### 5.2 结果可信度检查流程

用户打开报告后，应能快速回答：

1. 数据区间是否完整？
2. 是否使用了正确复权口径？
3. 成本、滑点和印花税是否启用？
4. 是否存在大量被 T+1、涨跌停或停牌拒绝的订单？
5. 收益是否集中在少数交易？
6. 回撤是否集中在某个市场阶段？
7. 换手率是否过高？
8. 策略相对基准是否真的有超额？
9. 参数是否可能过拟合？

### 5.3 策略对比流程

1. 用户选择多个回测结果。
2. 系统校准时间区间和基准。
3. 系统展示关键指标对比。
4. 系统标记：
   - 收益最高策略。
   - 回撤最低策略。
   - 夏普最高策略。
   - 成本最高策略。
   - 换手率最高策略。
   - 参数不稳定策略。
5. 用户选择候选策略进入更深归因。

## 6. 功能需求

### 6.1 数据服务模块

| 功能 | 说明 | 优先级 |
| --- | --- | --- |
| 数据源适配器 | 封装 AKShare 调用，避免业务层直接依赖外部接口 | P0 |
| 标的标准化 | 将用户输入转换为系统统一 symbol | P0 |
| 日线数据标准化 | 统一日期、开高低收、成交量、成交额、复权口径 | P0 |
| 本地缓存 | 将标准化数据保存为 Parquet 或 DuckDB | P0 |
| 增量更新 | 已缓存数据只补齐缺失区间 | P1 |
| 数据质量校验 | 检查重复日期、缺失值、非法价格、异常成交量 | P0 |
| 复权口径记录 | 每份数据记录不复权、前复权或后复权 | P0 |
| 多源备份 | 后续支持 Tushare 或其他数据源 | P2 |

设计原则：

- 回测引擎不能直接调用 AKShare。
- 外部数据必须先经过标准化和质量校验。
- 数据问题不能静默吞掉，必须出现在报告或运行日志中。

### 6.2 A 股规则模块

| 规则 | V0.1 行为 | 后续增强 |
| --- | --- | --- |
| T+1 | 当日买入的股份次一交易日才可卖 | 按 lot 维度跟踪可卖数量 |
| 整手交易 | 买入数量向下取整到 100 股 | 支持零股卖出清仓 |
| 手续费 | 按费率和最低收费计算 | 支持券商模板 |
| 印花税 | 卖出侧收取 | 支持历史税率变化 |
| 滑点 | 使用固定基点滑点 | 支持成交量和波动率相关滑点 |
| 涨跌停 | 用日线价格做近似限制 | 后续接入更可靠涨跌停数据 |
| 停复牌 | 停牌日不允许成交 | 接入显式停复牌数据 |

### 6.3 策略模块

策略模块负责表达交易意图，不负责撮合和成本。

策略输入：

- 标准化行情数据。
- 当前日期。
- 当前持仓。
- 可用现金。
- 策略参数。

策略输出：

- 买入/卖出信号。
- 目标仓位。
- 订单意图。

V0.1 内置策略：

- 双均线策略：
  - 短均线上穿长均线买入。
  - 短均线下穿长均线卖出。
  - 参数包括短周期、长周期、目标仓位、是否允许空仓。

### 6.4 回测引擎模块

回测引擎是核心执行层。

职责：

1. 读取回测配置。
2. 加载标准化行情数据。
3. 对齐交易日历。
4. 初始化现金、持仓和组合状态。
5. 按交易日推进事件循环。
6. 调用策略生成订单意图。
7. 调用 A 股规则模块校验订单。
8. 调用撮合模块生成成交或拒单。
9. 更新现金、持仓、可卖数量和组合净值。
10. 记录订单、成交、拒单、持仓和资金曲线。
11. 输出结构化回测结果。

引擎输出：

- 运行元数据。
- 每日净值曲线。
- 每日持仓。
- 订单记录。
- 成交记录。
- 拒单记录。
- 平仓交易记录。
- 成本汇总。

### 6.5 分析与归因模块

V0.1 指标：

- 累计收益。
- 年化收益。
- 年化波动率。
- 夏普比率。
- 最大回撤。
- 卡玛比率。
- 胜率。
- 盈亏比。
- 平均持仓周期。
- 换手率。
- 总手续费。
- 总印花税。
- 总滑点成本。

V0.5 指标：

- 相对基准收益。
- 超额收益。
- 跟踪误差。
- 信息比率。
- 滚动夏普。
- 滚动回撤。
- 月度收益热力图。
- 分阶段收益。

V1.0 归因：

- 选股贡献。
- 择时贡献。
- 行业贡献。
- 风格因子暴露。
- 因子 IC/IR。
- 因子分层收益。
- 因子衰减。

### 6.6 报告模块

V0.1 报告结构：

1. 运行摘要。
2. 策略配置。
3. 数据元信息。
4. 关键指标卡。
5. 净值曲线。
6. 回撤曲线。
7. 成本汇总。
8. 交易行为摘要。
9. 交易明细。
10. 拒单明细。
11. 数据和规则警告。
12. 配置附录。

警告必须显式展示：

- 存在缺失数据。
- 使用零成本或异常低成本。
- 基准区间和策略区间不一致。
- 换手率过高。
- 交易样本过少。
- 参数范围异常。
- 涨跌停模拟为近似模型。

## 7. 非功能需求

| 类别 | 要求 |
| --- | --- |
| 可复现性 | 每次运行保存配置、时间区间、数据源、复权方式、成本参数和生成时间 |
| 可审计性 | 每笔拒单必须有 reason code 和说明 |
| 性能 | V0.1 单标的日线回测应能在本地交互式完成 |
| 可维护性 | 数据、策略、规则、引擎、分析、报告分层清晰 |
| 可扩展性 | 新策略和新数据源不应修改引擎核心 |
| 可靠性 | 数据质量问题在回测前或报告中暴露 |
| 安全边界 | 报告明确说明回测不代表未来收益，不构成投资建议 |

## 8. 架构设计

### 8.1 分层架构

```text
┌──────────────────────────────────────────────────────────────┐
│ 表现层                                                       │
│ CLI 报告入口 / 后续 Streamlit 或 Dash 工作台                  │
├──────────────────────────────────────────────────────────────┤
│ 报告与分析层                                                 │
│ 指标计算 / 回撤 / 交易行为 / 归因 / 报告导出                  │
├──────────────────────────────────────────────────────────────┤
│ 回测引擎层                                                   │
│ 事件循环 / 组合状态 / 模拟 broker / 成交记录                  │
├──────────────────────────────────────────────────────────────┤
│ A 股规则层                                                   │
│ T+1 / 整手 / 成本 / 滑点 / 涨跌停 / 停复牌                    │
├──────────────────────────────────────────────────────────────┤
│ 策略层                                                       │
│ 策略模板 / 信号生成 / 目标仓位                                │
├──────────────────────────────────────────────────────────────┤
│ 数据服务层                                                   │
│ 数据源适配 / 标准化 / 校验 / 缓存                             │
├──────────────────────────────────────────────────────────────┤
│ 数据源层                                                     │
│ AKShare 主源 / 后续 Tushare 或其他备源                        │
└──────────────────────────────────────────────────────────────┘
```

### 8.2 模块职责

| 模块 | 负责 | 不负责 |
| --- | --- | --- |
| 数据适配器 | 拉取外部原始数据 | 策略逻辑和撮合 |
| 数据标准化 | 统一字段、日期、复权口径 | 隐藏数据质量问题 |
| 数据缓存 | 保存和读取标准化数据 | 修改交易规则 |
| 策略模块 | 生成信号或目标仓位 | 计算手续费和执行成交 |
| 规则模块 | 校验订单是否符合 A 股规则 | 判断策略是否应该买卖 |
| 撮合模块 | 生成成交、拒单和资金变化 | 访问外部数据源 |
| 分析模块 | 从回测结果计算指标 | 修改订单或成交 |
| 报告模块 | 格式化输出给用户 | 重新计算撮合结果 |

### 8.3 数据流

```text
用户配置
  -> 数据请求
  -> 本地缓存查询
  -> 缺失数据拉取
  -> 数据标准化
  -> 数据质量校验
  -> 策略信号生成
  -> 订单意图生成
  -> A 股规则校验
  -> 模拟成交或拒单
  -> 组合状态更新
  -> 指标计算
  -> 报告导出
```

symbol: 系统统一标的代码

end_date: 结束日期，包含
frequency: V0.1 固定为 daily
adjustment: none / qfq / hfq
source: 默认 akshare
```

```

### 9.3 策略配置

```text
strategy_name
parameters

```

### 9.4 订单

```text
order_id
date

```

### 9.5 成交

```text
fill_id

date

```

### 9.6 拒单

```text
order_id
date
symbol
side
requested_quantity
reason_code
reason_message
```

推荐拒单原因：

- `INSUFFICIENT_CASH`：现金不足。
- `INSUFFICIENT_POSITION`：持仓不足。
- `T_PLUS_ONE_RESTRICTED`：T+1 限制。
- `LOT_SIZE_TOO_SMALL`：不足一手。
- `LIMIT_UP_BUY_BLOCKED`：涨停买入受限。
- `LIMIT_DOWN_SELL_BLOCKED`：跌停卖出受限。
- `HALTED`：停牌。
- `MISSING_BAR`：缺少行情数据。

## 10. 数据设计

### 10.1 本地存储方案

V0.1 推荐：

- 标准化日线数据使用 Parquet。
- 回测配置使用 YAML 或 JSON。
- 回测输出使用 CSV 或 Parquet。
- 报告使用 Markdown 或 HTML。

V0.5 推荐：

- 使用 DuckDB 查询本地数据。
- Parquet 继续作为列式持久化格式。

### 10.3 数据质量规则

P0 校验项：

1. 同一标的的日期唯一。
2. 日期升序排列。
3. 开高低收为正数。
4. 最高价大于等于最低价。
5. 成交量非负。
6. 复权方式被记录。
7. 缺失交易日被记录。
8. 数据源和获取时间被记录。

处理策略：

- 价格字段非法时应终止回测。
- 缺失数据可允许继续，但必须进入警告区。
- 数据标准化不能静默修正核心字段，所有修正应进入元数据。

## 11. A 股规则建模

### 11.1 T+1

V0.1 需要跟踪“可卖数量”。T 日买入的股份在 T+1 交易日才进入可卖数量。

默认行为：

- 卖出数量超过可卖数量时拒绝超出部分。
- 记录拒单原因。
- 不允许策略绕过 T+1。

### 11.2 整手交易

买入：

- 数量向下取整到 100 股。
- 取整后为 0 时拒单。

卖出：

- V0.1 同样按 100 股处理。
- 后续版本可支持零股清仓。

### 11.3 成本

V0.1 成本模型：

- 手续费率可配置。
- 最低手续费可配置。
- 印花税仅卖出侧收取。
- 滑点按固定基点配置。

报告要求：

- 必须展示手续费、印花税、滑点成本合计。
- 零成本回测必须标记为警告。

### 11.4 涨跌停

日线数据无法准确模拟盘口排队，因此 V0.1 使用保守近似：

- 如果模拟成交价格处于涨停价，买入可被阻止。
- 如果模拟成交价格处于跌停价，卖出可被阻止。
- 如果没有可靠涨跌停字段，报告必须说明该模拟是近似。

### 11.5 停复牌

若没有显式停牌数据：

- 缺失 bar 的日期不允许成交。
- 持仓估值可使用上一有效收盘价。
- 报告必须披露这一近似。

## 12. 报告设计

### 12.1 V0.1 报告版式

1. 标题区：
   - run id。
   - 策略名称。
   - 标的。
   - 日期区间。
   - 基准。
   - 生成时间。

3. 净值曲线。
4. 回撤曲线。
5. 成本汇总。
6. 交易行为摘要。
7. 订单与成交表。
8. 拒单表。
9. 数据质量警告。
10. 配置附录。

### 12.2 报告应回答的问题

- 策略是否获得正收益？
- 策略是否跑赢基准？
- 最大回撤是否可接受？
- 收益是否集中于少数交易？
- 成本是否显著侵蚀收益？
- 换手率是否过高？
- 是否有大量订单被 A 股规则拒绝？
- 数据质量是否影响结论？

## 17. 待决策事项

### 17.1 是否直接使用 Backtrader

推荐：用 Backtrader，但在外层保留内部接口。

原因：

- 早期可以利用成熟事件驱动能力。
- 内部接口能避免未来替换引擎时影响策略、数据和报告层。

### 17.2 V0.1 是否做 UI

推荐：先 CLI + 报告导出，不做复杂 UI。

原因：

- 回测可信度比界面更重要。
- CLI 更容易测试和复现。
- Streamlit 可在 V0.5 添加。

### 17.3 缓存使用 Parquet 还是 DuckDB

推荐：V0.1 使用 Parquet，V0.5 加 DuckDB。

原因：

- Parquet 简单，适合快速落地。
- DuckDB 更适合多标的、多条件查询，等数据规模上来后再引入。

### 17.4 涨跌停是否必须进入 V0.1

推荐：进入 V0.1，但标记为近似模拟。

原因：

- 涨停买不进、跌停卖不出对 A 股策略影响很大。
- 但日线数据天然无法精确模拟排队，因此必须在报告中披露限制。

### 17.5 因子归因是否进入 V0.1

推荐：不进入 V0.1。

原因：

- 因子归因依赖更复杂的数据结构和组合回测。
- V0.1 应先保证核心回测链路可靠。

## 19. 本设计内置的产品决策

1. 可信度优先于功能数量。
2. A 股规则是引擎约束，不是报告注释。
3. 数据质量必须对用户可见。

6. 归因分析是长期差异化，但不阻塞 MVP。
7. 所有结果都应能被配置、数据、订单和成交记录解释。
